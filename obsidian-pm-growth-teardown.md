## 一、技术栈拆解

### 1.1 主题（Theme）

从截图 UI 风格判断，使用的是 **Blue Topaz** 主题（或基于 Blue Topaz 的定制主题）。

识别依据：
- 左侧栏的浅蓝色调和圆角文件夹图标
- 右侧面板的卡片式布局和进度条样式
- 顶部标签页的设计风格
- 整体配色与排版是 Blue Topaz 的典型特征

| 项目 | 推荐配置 |
|------|----------|
| 主题 | **Blue Topaz**（社区插件商店安装） |
| 辅助插件 | **Style Settings**（用于微调 Blue Topaz 的各项样式参数） |

### 1.2 已识别的插件清单

从截图左侧图标栏、底部状态栏、右侧面板内容推断：

| 插件 | 识别依据 | 用途 |
|------|----------|------|
| **Copilot** | 左侧栏有 `copilot` 文件夹 | AI 对话助手，集成在 Obsidian 内部 |
| **Dataview** | 右侧面板的结构化属性展示 | 查询笔记元数据，生成动态视图 |
| **Templater** | 规范化的模板结构和变量 | 高级模板引擎 |
| **Spaced Repetition** | 底部状态栏显示 `0 card(s) due / 0 note(s) due` | 间隔重复学习（复习卡片） |
| **Calendar** | 配合每日笔记使用 | 日历导航 |
| **Style Settings** | 右侧面板的自定义样式（进度条、卡片） | 微调主题参数 |
| **Banners / Custom Frames** | 右侧个人资料卡片的头像区域 | 笔记顶部横幅或自定义卡片 |
| **Kanban**（推测） | 有 `Week 4 todo list` / `My Task` 等任务管理笔记 | 看板式任务管理 |

### 1.3 CSS Snippets（自定义样式）

右侧面板的「属性打分卡片」使用了 **CSS Snippet（自定义 CSS 代码片段）** 来实现：
- 头像圆形裁切 + 居中展示

- 各技能维度的彩色进度条（不同颜色对应不同等级）

**启用方式**：`设置 → 外观 → CSS 代码片段 → 打开代码片段文件夹 → 放入 .css 文件`

### 2.2 结构设计理念

这套结构不是传统的 PARA，而是一种 **「输入 → 业务 → 体系」三层递进模型**：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  1. 原始输入   │ →  │ 2. 业务知识   │ →  │ 3. 知识体系   │
│  (Raw Input)  │    │ (Domain)     │    │ (System)     │
├──────────────┤    ├──────────────┤    ├──────────────┤
│ · 未分类       │    │ · Feelora    │    │ · 需求分析    │
│ · 工作日志     │    │   产品的业务   │    │ · AI产品      │
│ · 产品思考     │    │   知识沉淀    │    │ · 数据分析    │
│ · 周报        │    │              │    │ · 商业认知    │
│ · 需求文档     │    │              │    │ · ...        │
│ · 用研与竞品   │    │              │    │              │
└──────────────┘    └──────────────┘    └──────────────┘
   信息收集层            知识加工层            能力沉淀层
```

**核心逻辑**：
- **第 1 层**（原始输入）：日常工作中产生的所有信息，按类型粗分（工作日志、产品思考、需求文档等），未分类的先进「未分类」文件夹
- **第 2 层**（业务知识）：从原始输入中提炼出的、与当前业务（Feelora）强相关的知识
- **第 3 层**（知识体系）：按产品经理能力模型组织的系统化知识，每个子文件夹 = 一个能力维度

### 2.3 每个能力维度的内部结构

从截图可见，第 3 层的每个子文件夹内部采用统一的组织方式：

```
{能力维度}/
├── 1. 说明             # 总览：该维度包含哪些方法和框架
├── 具体知识笔记 A       # 某个具体方法论或框架的详细笔记
├── 具体知识笔记 B
└── ...
```

**「说明」笔记的作用**：充当该维度的 **MOC（Map of Content 内容地图）**，列出本维度涵盖的所有方法与框架，并通过双向链接关联到具体笔记。

### 3.4 校准机制

截图中可见**带时间戳的校准记录**，这是一个关键设计：

```
2026-04-19  第一次定版
  初始 Lv.7  多维竞品追赶分析一项竞赛，主动泛化 AI 在占卜学术论文，组从人
  按照时间「尽在转转时间向简竹/慕斯信息真正用」框架，对1.5 PM 经验
  的研究范式进行水平的完整评估

2026-04-20  分数校准（自我纠正）
  Lv.7→6  证据原是实说，但在绝对尺度上 Lv.7 代表的是能影响洞
  到研究范式的水平，目前证据不够支撑
```

**校准逻辑**：
1. **初始评分**：基于自身经历和能力证据，给出初始等级
2. **定版记录**：写明评分依据（哪些事实支撑这个等级）
3. **自我纠正**：重新审视后调整，并记录调整原因
4. **时间戳追踪**：每次变更都标注日期，形成成长时间线

```markdown
| Lv.1 | 了解 | 知道这个领域，但没有实际经验 |
| Lv.2 | 入门 | 有过 1-2 次实践，需要大量指导 |
| Lv.3 | 基础 | 能在指导下完成工作，理解基本方法 |
| Lv.4 | 熟练 | 能独立完成常规任务，掌握主要方法论 |
| Lv.5 | 精通 | 能高质量独立完成，处理复杂场景 |
| Lv.6 | 专业 | 能输出方法论，指导他人，有体系化认知 |
| Lv.7 | 专家 | 行业标杆水平，能创造新范式 |
```

**评分三原则**（从截图的校准记录中提炼）：
1. **证据驱动**：每个等级必须有事实支撑，不凭感觉打分
2. **绝对尺度**：参照行业标准，而非只和自己比
3. **动态校准**：定期重新审视，允许降级和升级

### 4.6 第 5 步：持续运营

| 频率 | 动作 |
|------|------|
| 每天 | 工作日志记录到 `1.原始输入/工作日志/` |
| 每周 | 整理未分类笔记，归入正确位置 |
| 每月 | 审视能力等级，看是否有新的证据支撑调整 |
| 每季度 | 全面校准能力评分，更新成长路线图 |

## 五、关键设计亮点总结

| 设计点 | 价值 |
|--------|------|
| **三层递进结构** | 信息自然从「收集→加工→沉淀」流动，避免一次性分类的心智负担 |
| **能力维度 = 文件夹** | 知识体系和能力模型一一映射，学什么直接对应补什么短板 |
| **每个维度有「说明」** | MOC 模式，快速纵览该维度全貌 |
| **带证据的打分** | 不是虚浮的自评，而是有事实支撑的量化评估 |
| **校准机制** | 内置纠偏，防止自我膨胀或过度自谦 |
| **时间戳记录** | 形成成长时间线，回看进步轨迹 |
| **Copilot 集成** | AI 辅助知识加工，提升效率 |
| **间隔复习** | Spaced Repetition 确保知识不只是存着，还真正内化 |
