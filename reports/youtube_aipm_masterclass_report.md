## 一、AI PM 到底是不是噱头？

**结论：不是，但要看你指的是哪类 AI PM。**

Jyothi 基于她在 Meta、Netflix、Etsy 的招聘经历，将市场上的"AI PM"岗位分为两大类：

### 类型一：传统 PM + AI 功能（占市场约 80%）
- 核心产品在 AI 出现之前就存在
- AI 是"加装"在现有产品上的功能
- 典型例子：在客服系统加聊天机器人；在文档工具加 AI 摘要
- **好消息：这类岗位数量是 AI 原生岗位的 4 倍，转型门槛相对低**

### 类型二：AI 原生 PM（占市场约 20%）
- 产品本身就是 AI，去掉 AI 产品就不存在了
- 典型例子：ChatGPT、GitHub Copilot、Claude、Cursor、Perplexity
- 产品本质上是**概率性的**（probabilistic），相同输入可产生不同输出
- **难度更高，但成长空间更大**

## 二、AI PM 的职级层级（技术栈层次）

Jyothi 将 AI PM 的职位按技术栈由上到下分为三层：

| 层级 | 核心职责 | 岗位占比 | 入门难度 |
|------|---------|---------|---------|
| **应用层 PM** | 拥有端到端用户体验，关注人机交互、信任建立、可靠性 | ~60% | ⭐（最容易） |
| **平台层 PM** | 为其他开发团队构建工具，包括开发者平台、模型编排、评估框架 | ~30% | ⭐⭐ |
| **基础设施层 PM** | 构建底层系统，如向量数据库、GPU 编排、模型推理优化 | ~10% | ⭐⭐⭐（最难） |

> 规律：越往下，技术深度要求越高，岗位数量越少。**想转型的人从应用层入手最合适。**

## 三、AI PM 必须掌握的 5 大核心概念

视频以系统性"大师课"的形式覆盖了以下五大主题：

### 概念一：AI PM 与普通 PM 的本质区别

**传统产品是确定性的（deterministic）：**
- 点击同一个按钮，永远跳转同一个页面
- 输入确定 → 输出确定

**AI 产品是概率性的（probabilistic）：**
- 相同输入可能产生不同输出
- PM 要以"质量分布"和"可接受错误率"来思考，而不是"成功 vs 失败"的二元思维

作为 AI PM，你需要回答的问题从"功能能不能用"变成：
- 用户在什么错误率下会失去信任？
- 发生 5% 概率的边缘情况如何处理？
- 是否需要备用的确定性系统？

**另一个核心区别：数据是第一公民。**
- 传统 PM 聚焦功能和用户流程
- AI PM 必须把数据当产品体验来设计
- 低质量数据 = 低质量输出（垃圾进，垃圾出）

此外，AI 产品的单位经济模型也不同于传统产品——成本是可变的，取决于 LLM 的响应长度。

### 概念二：什么时候该用 AI，什么时候不该用

**MIT 研究显示：19/20 的 AI 试点项目失败，关键原因是选错了问题。**

#### ✅ 适合用 AI 的场景

| 场景 | 说明 | 案例 |
|------|------|------|
| 复杂模式识别 | 规律存在但人工难以定义规则 | YouTube 内容推荐（用户行为与内容偏好关系多维） |
| 历史数据预测 | 基于多年历史数据预测未来 | Amazon 库存需求预测（综合季节性、促销、天气等上百个变量） |
| 大规模个性化 | 为百万级用户提供个性化体验 | 内容推荐引擎 |

#### ❌ 规则/启发式方法（heuristics）已经足够的场景

| 场景 | 原因 |
|------|------|
| 可解释性不可妥协（医疗、金融监管等） | AI 模型的可解释性依然较弱 |
| 领域规则明确且全面 | 如税务计算软件，逻辑是显式的 |
| 历史数据不足 | AI 需要大量数据才能有效，新市场/新功能不适合 |
| 上市时间极其紧张 | AI 系统开发周期通常更长 |

### 概念三：如何选择合适的 AI 技术

很多人遇到问题就直接想到"用 LLM"，但 Jyothi 提出：**一个简单的机器学习模型可能一周就能解决，且成本只有几分之一。**

她将 AI 技术分为三个桶：

#### 桶 1：传统机器学习（Traditional ML）
- **代表技术：** 回归、随机森林、XGBoost
- **适用条件：** 结构化数据，预测或分类任务
- **判断问题：** 能不能把这个问题放进一张电子表格（有明确输入列和要预测的输出列）？
- **典型用例：** 欺诈检测、客户流失预测
- **优势：** 成熟、可靠、解释性强、速度快

#### 桶 2：深度学习（Deep Learning）
- **代表技术：** 神经网络、计算机视觉、语音识别
- **适用条件：** 图像/视频/音频等非结构化感知类数据
- **判断问题：** 这是感知任务吗？人类能做但写不出规则吗？
- **典型用例：** 医疗图像诊断、工厂生产线缺陷检测、语音助手
- **代价：** 需要更多数据、更多算力，可解释性低于传统 ML

#### 桶 3：生成式 AI（Gen AI）
- **代表技术：** LLM（Claude、ChatGPT）、Stable Diffusion 等扩散模型
- **适用条件：** 需要理解、生成或推理自然语言/图像
- **判断问题：** 任务需要读写自然语言吗？需要常识推理吗？用户会用对话形式交互吗？
- **典型用例：** 对话界面、内容生成、跨来源综合推理

> **关键思维：** 这三类技术不是竞争关系，是工具箱里的不同工具。最好的 AI 产品通常组合使用多种方法。

### 概念四：AI Agent、提示工程与 RAG

#### 4.1 什么是 AI Agent？

AI Agent（代理）是能够自主决策、采取行动以实现某个目标的系统。核心特征是**目标导向**——你不告诉它执行顺序，它自己推理出路径。

**AI Agent 的四个构建模块：**

```
┌─────────────────────────────────────────────────────┐
│  1. 感知（Perception）：接收输入（文本/图像/传感器/API）  │
│  2. 推理（Reasoning）：LLM/分类模型/规划算法在此处运行   │
│  3. 执行（Execution）：生成文本/调用 API/控制硬件        │
│  4. 学习（Learning）：评估结果并持续改进               │
└─────────────────────────────────────────────────────┘
```

#### 4.2 工作流 vs. Agent：关键区别

| 维度 | 工作流（Workflow） | Agent（代理） |
|------|-----------------|-------------|
| 决策模式 | 预定义步骤，确定性 | 自主决定用哪些工具 |
| 控制方式 | 人工定义决策树 | 目标驱动，自主规划 |
| 适用场景 | 发票处理、固定流程自动化 | 开放式问题解决 |

**N8N 实操对比演示（视频现场演示）：**

> 同样的天气查询 + Gmail 发送邮件任务：
> - **工作流版本：** 手动写代码格式化数据，按顺序调用两个 API，步骤固定
> - **Agent 版本：** 只给 Agent 两个工具（天气 API + Gmail），让它自行决定调用哪个
>   - 当问"今天洛杉矶天气如何"→ Agent 只调用了天气 API（没有执行 Gmail）
>   - 当说"发送这条消息"→ Agent 才调用 Gmail
>   - **没有人告诉它什么时候用哪个工具，它自己判断的**

#### 4.3 提示工程（Prompt Engineering）

提示词是你与 AI 系统的主接口，决定 Agent 的行为和输出质量。

**三类提示词：**
- **系统提示词（System Prompt）：** 设定 Agent 的整体行为、人格和规范（如：客服 Agent 的同理心、专业语气、身份验证规则）
- **用户提示词（User Prompt）：** 用户实际输入的内容
- **少样本示例（Few-shot Examples）：** 提供好/坏回答的案例，让模型知道什么是目标输出

> 被严重低估的技巧：少样本示例在生产系统中极其强大。

#### 4.4 上下文工程（Context Engineering）

这是生产环境中的核心技能。

AI 模型有**上下文窗口限制**（Claude Sonnet 为 200K tokens）。听起来很多，但加载公司知识库 + 对话历史 + 实时数据后，很快就会触顶，此时你必须做出取舍：哪些内容放进来，哪些不放。

**上下文的三个层次：**
1. **即时上下文（Immediate Context）：** 当前对话或任务
2. **会话上下文（Session Context）：** 用户近期交互历史
3. **知识上下文（Knowledge Context）：** Agent 需要参考的更广泛知识库

> 每个 token 都要花钱。随意把整个知识库加载到每次交互中是在烧钱。上下文工程是"知道加载什么、什么时候加载"的艺术。

#### 4.5 RAG（检索增强生成）系统

RAG 是解决"动态上下文管理"问题的核心技术方案。

**RAG 解决什么问题？**  
LLM 的训练数据有截止日期，且无法直接访问企业私有数据。RAG 让模型能"查阅"最新的或企业专属的文档来回答问题。

**RAG 工作原理（简化版）：**

```
文档 → 分块（Chunking）→ 向量化（Embedding）→ 向量数据库

用户问题 → 向量化 → 在向量数据库中查找最相近的文档块
          ↓
  【相关文档块 + 用户问题】→ LLM → 基于文档内容的回答
```

**分块策略举例：** 把一本书每 20 页撕成一块，类似固定长度分块策略。

**Langflow 实操演示（视频现场构建）：**
1. 上传"2025年 AI 商业现状"报告
2. 用分割节点切块
3. 用 OpenAI embedding-3-small 模型向量化
4. 存入 Astra DB 向量数据库
5. 用户提问 → 检索相关块 → 加入提示词模板 → LLM 生成回答

**关键建议——Fine-Tuning 的使用顺序：**

> 很多人看到 Fine-tuning 在 API 文档里就直接跳到它。**错误！**

正确顺序：
1. 提示词优化（Prompt Optimization）
2. 上下文工程（Context Engineering）
3. RAG（检索增强生成）
4. **最后才考虑 Fine-tuning**

> **80% 的用例用 RAG 就能解决，根本不需要 Fine-tuning。**

### 概念五：如何交付 AI 产品（从项目到产品）

**Jyothi 的核心建议：不要把它当"课程项目"，要把它当"真实产品"来做。**

**分步骤行动路径：**
1. 找到你自己的真实痛点，而不是随便选一个题目
2. 构建出来
3. **发布它**——让朋友、家人来用
4. 遇到真实用户 → 遇到真实崩溃 → 像真正的 PM 一样解决真实问题
5. 拥有可以在简历中说出"这个 bug 在这些场景下出现，我是这样修复的"的真实素材

**理想作品集结构：**
- 一个解决真实问题的 App（全流程：构建 → 发布 → 收到真实反馈）
- 一个你构建的 AI Agent
- 一个 RAG 系统

**关于证书：**
> 证书不是核心，但能作为"已获认可的信号"向招聘方传达你的学习成果是可信的。推荐路径：学完后考 AWS AI Practitioner 认证。

## 四、大厂 AI PM 文化对比

这是视频中很有价值的部分：Jyothi 亲历了 Amazon、Meta、Netflix 三家公司的产品文化，给出了第一手对比。

### 4.1 Amazon / AWS：文档驱动，客户执念

- 核心方法论：**PRFAQ（新闻稿 + 常见问题）**
- 在写第一行代码或设计稿之前，必须先写一份"如果这个产品上线了，记者会怎么报道"的新闻稿
- 文档会一路审到 VP 乃至 Andy Jassy
- PM 有 40–50% 的时间在写文档
- **核心理念：** 从客户问题出发，倒推解决方案
- **你能得到的：** 成为一个极其出色的写作者

### 4.2 Meta：数据驱动，快速迭代

- 和 Amazon 恰好相反：极简流程，追求速度
- PM 被期望非常懂技术——能看代码、追 insight、讨论实现细节
- 同时运行多个变体，让数据决定哪个更好
- **核心理念：** 快速行动，实验先行，数据说话
- **Meta 拥有业内最复杂的实验基础设施**
- **你能得到的：** 极强的 A/B 测试和统计显著性思维

### 4.3 Netflix：语境驱动，高度自主

- 既不是 Amazon 的严格文档，也不是 Meta 的高速实验
- **核心理念：** context over control（语境优于控制）
- 投入大量精力确保每个人理解战略方向，然后充分信任你独立决策
- 期望你是极强的沟通者，能在模糊中定义自己的工作边界
- **你能得到的：** 真正理解自主权的含义，以及如何在共识中构建影响力
