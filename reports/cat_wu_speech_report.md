## 二、核心观点：旧方法的前提不成立了

### 2.1 传统 PM 节奏为何失效？

传统产品经理的工作有一个隐含前提：  
> "项目开始时的技术边界，和项目结束时差不多。"

因此过去 PM 可以安心地用 6–12 个月制定路线图、写 PRD、按计划执行。

**但 AI 模型的能力在指数级增长，这个前提已被打破。**  
你刚开始设计方案时绕开的"技术限制"，可能项目做到一半就已经不存在了。你是在一块持续升高的地面上盖楼。

### 2.2 她用一个测试浓缩了这一洞察

Cat Wu 做了一个持续超过一年的基准测试：  
> 每当 Anthropic 发布新模型，她就让 Claude Code 为开源画板工具 **Excalidraw 添加表格功能**。

| 时间 | 模型版本 | 结果 |
|------|---------|------|
| 2024年10月 | Sonnet 3.5 (new) | 做到一半卡住，无法完成 |
| 2025年6月 | Opus 4 | 偶尔能跑通，录屏作为发布会 demo |
| 2026年（约） | Opus 4.6 | 足够稳定，在数千名开发者面前直播演示 |

这个测试说明：模型能力在 16 个月内提升了约 **41 倍**（METR评估机构数据：Opus 4.6 有一半概率完成人类需要 12 小时的软件任务；而 16 个月前 Sonnet 3.5 只能完成 21 分钟级别的任务）。

## 三、新时代的产品节奏：三个关键词

Cat Wu 给出的新节奏只有三句话：

```
快速实验 → 持续交付 → 加倍押注有效方向
```

- **快速实验**：对可能性保持好奇心，把探索成本降到最低。
- **持续交付**：将发布流程做成可重复机制，消除每次发布的协调摩擦。
- **加倍押注**：找到真实有效的方向后，快速放大投入。

她在 Anthropic 实现了将"从想法到用户手中"的最短周期压缩至 **1 天**。

## 四、AI 时代 PM 的四条核心做法

### 4.1 鼓励"side quest"，别锁死路线图

传统做法是：先探索 → 锁路线图 → 执行。  
Claude Code 团队反过来：**鼓励所有人在正式路线图之外做"side quest"（旁线探索）。**

- 花一个下午原型化一个想法
- 测试一个你以为做不到的功能
- 把模型推到极限看看会发生什么

> **结果：** 桌面版 Claude Code、AskUserQuestion 工具、待办清单（/todo），这些 Anthropic 最受欢迎的功能，全部来自 side quest，而非规划中的路线图。

### 4.2 先出 Demo，后写文档

团队不开传统站会，改成**分享 Demo 会**。

- 先有内部用户真实试用
- 有人用、用得好的功能，才继续打磨推广
- 一个下午做原型，押错了损失极小

**实际案例：**  
工程师 Noah 写完插件规范，直接交给 Claude Code 生成原型。出来的东西接近生产可用。这个原型直接锚定了最终方案，因为团队拿它验证了用户体验，而不是对着文档争论。

> **Cat 的建议：** 写完规范后，先把它发给 Claude Code，让它试着做一遍。哪怕出来的很粗糙，也比空聊有用得多。

### 4.3 每次模型升级，回头看一遍老功能

新模型出来是一个提醒：你过去受限于模型能力而妥协的功能，现在可能一夜之间变得好用。

- 主动当日活用户，故意让模型做你觉得太难的事
- 成功了，说明产品要跟上了

**实际案例：**  
Chrome 集成功能的诞生：团队发现用户在 Claude Code 里写完 Web 应用后，会手动打开 Chrome 里的 Claude 来测试，在两个工具之间复制粘贴。当手动流程跑通，这就说明这件事该做成内置功能。

另一个技巧：  
> 先验证能力上限，别急着省 token。过早做成本优化是常见错误，你会交付一个"打了折的东西"。等便宜模型追上来再优化成本，但前提是你得先知道这功能能不能成立。

### 4.4 越简单越好

Anthropic 内部核心原则：**做简单的、能用的事。**

如果你为旧模型写了一个巧妙的 workaround，下一代模型一出，这个补丁可能就是负担。实现越简单，新能力来的时候替换越轻松。

**待办清单的进化：**

| 阶段 | 做法 | 问题 |
|------|------|------|
| 初版 | 在系统提示里加定期提醒，让 agent 每隔几条消息检查待办状态 | 能用，但是补丁 |
| 新模型发布后 | 新模型原生支持该行为，提醒被直接删除 | 简洁了 |

Opus 4.6 的系统提示词比之前少了 **20%**，正是这种"随模型进步精简 workaround"的自然结果。

## 五、PM 角色与团队文化的变化

### 5.1 角色边界正在融合

在 Claude Code 团队：
- **设计师**开始交付代码
- **工程师**参与产品决策
- **PM 自己**搭原型跑评估

这之所以行得通，是因为**战略目标足够清晰**，每个人可以自主判断优先级，不需要层层审批。

### 5.2 "产品品味"成为最稀缺的技能

当代码生成成本趋近于零，差异化不在"怎么写代码"，而在"写什么"。

> "决定要做什么，比知道怎么做更有价值。产品品味是一种非常稀缺的技能。"

### 5.3 拥抱混乱的团队文化

Anthropic 招聘时特别看重：  
对自己**精力管理**有深刻理解的人——能在持续的高频变化中不被"燃尽"。

她把工作节奏比作不断升级的紧急事项（P0 → P00 → P000），并说：
> "如果你没有这种心态，你很快就会被燃尽。"

## 六、PM 的日常工具组合（Cat Wu 本人实践）

Cat Wu 日常将三个工具分工协作，完成了几百小时的工作，零行手写代码：

| 工具 | 用途 |
|------|------|
| **Claude.ai** | 思考搭子。不需要执行动作时用——策略讨论、棘手问题、快速问答 |
| **Claude Code** | 写代码。原型、评估脚本、调 API |
| **Cowork** | 剩下所有事——收件箱、待办、幻灯片、搜 Slack 找历史决策、订差旅 |

**Cowork 实战案例：** 准备一场大会演讲时，她通过 Cowork 连接 Slack、Google Drive 等数据源，让 AI 代理自行研究并生成了一份 20 页的幻灯片初稿。

## 七、Claude Code 的研发与增长哲学

### 7.1 工具先跑起来，规范后面补

Claude Code 的起点不是战略规划，而是工程师 Boris 把模型整合进自己日常工具的一次探索。Cat Wu 作为早期用户给了大量反馈，后来全职加入。

**成长路径：** 先在团队内部跑起来 → 整个 Anthropic 内部使用 → 研究员、数据科学家、PM、设计师都用上了 → 再向外部推广。

> **效果：** 自 2025 年 5 月以来，Claude Code 活跃用户增长近 **3 倍**，年化营收提升 **5.5 倍**。

### 7.2 深度内部试用（Dogfooding）文化

流程不是写 PRD → 评审 → 研发，而是：
1. 工程师直接用 Claude Code 快速做原型
2. 原型在公司内部（约 1000 人）发布试用
3. 反响热烈 → 正式推出；使用率低 → 重新审视

反馈来源：Anthropic 内部 Slack 聊天室，约 1000 人，每 10 分钟就有新反馈，质量非常高。

### 7.3 用 Claude Code 来改进 Claude Code

- Slack 反馈自动汇总
- GitHub 问题自动去重
- 文档更新先由 Claude Code 写第一版，再人工校对 10%

### 7.4 向外构建生态：SDK 平台

Claude Code 不只是命令行工具，而是一套**代理（Agent）构建平台**：

- 核心代理循环（处理对话轮次和工具调用）
- 内置权限系统
- 智能处理 API 错误、提示缓存节省成本
- 高度可定制：替换系统提示、接入自有工具

> 使用 SDK 约 **30 分钟**即可搭建出功能完整的代理原型。已有外部公司用此构建法律代理、合规代理等。

## 八、给 PM 的可执行建议（直接来自 Cat Wu）

1. **写完规范就跑一遍**：先把需求规格交给 Claude Code 试做，哪怕结果很粗糙，也比空谈有用。
2. **用 Demo 代替争论**：让原型替你说话，团队讨论的基础更具体。
3. **主动当你产品的日活用户**：这是感知模型能力边界最直接的方式。
4. **找出不能妥协的点，其他的松开**：放手才能跑快，完美主义是 AI 产品 PM 最大的阻力。
5. **每次模型发布都回头看旧功能**：新模型是提醒，不是干扰。
6. **验证能力上限优先于压缩成本**：先确认"这件事能成立"，再谈省钱。
7. **把你的 CLAUDE.md 维护好**：花时间写清楚架构、常见陷阱和团队习惯，能显著提升 Claude 输出质量。
